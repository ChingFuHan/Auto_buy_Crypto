- **適用範圍：** 所有專案、所有 Agent、所有協作者
- **優先級：** 本文件凌駕於任何子專案規範之上
- **版本：** v1.5.0
- **最後更新：** 2026-04-25

---

## 規則優先級與衝突處理

在規則衝突時，依下列優先級處理：

| 優先級 | 說明 |
|---|---|
| P0 - 絕對禁止 | 機密外洩、無限重試、未留快照就做破壞性操作、覆蓋歷史結果、Review 任務越界執行、未即時存檔導致對話成果遺失 |
| P1 - 高度重要 | Handoff、Local-first、版本快照、明確風險標示、自我驗證、Review/Execution 邊界 |
| P2 - 建議遵守 | 其餘工程慣例與專案最佳實踐 |

**衝突處理原則**

1. 下層規則不得違反、忽略、弱化或覆寫上層規則。
2. 若 `AGENTS.md`、`README.md`、`HANDOFF.md`、`final_file.md`、任務 prompt 或其他子專案文件與本文件衝突，必須先明確列出衝突點，再以本文件為準。
3. 若需例外，必須在 `HANDOFF.md` 以 `[EXCEPTION]` 記錄原因、影響與預計恢復時間。
4. 未記錄的例外，視為違規。

## RULE 01 | 智慧傳承與脈絡管理 - Handoff Protocol

對話視窗一旦關閉，未存檔內容永久消失。這是不可接受的損失。
本規則的核心目的是：**確保任何一次對話的成果，不會因為視窗關閉、平台斷線、context 重置而消失。**

### 強制存檔時機(P0 級)

Agent 必須在以下任一時機**立即**寫入 `HANDOFF.md`，不得延後：

1. 每個子任務完成時
2. 每個關鍵決策做出時(即使整體任務還沒結束)
3. 每個卡點被解開時(連同解法一併寫入)
4. 使用者出現「告一段落」訊號時，例如：
   - 「先這樣」
   - 「我先去 X」
   - 「明天繼續」
   - 「我要關了」
5. 長對話累積到 20 輪互動以上時，至少存檔一次
6. Agent 自行判斷對話可能即將結束時(例如使用者長時間未回應、語氣收尾)

### 存檔義務的剛性

- 不可因為「等等再寫一次比較完整」而延後
- 不可因為「這個決策可能還會變」而不寫(可標 `[TENTATIVE]`)
- 不可只在腦中記住，不寫進檔
- 不可假設使用者會自己記得；使用者來這裡就是為了不必記得
- 違反存檔義務 = P0 違規

### 雙檔架構

| 檔案 | 用途 | 大小限制 | Agent 行為 |
|---|---|---|---|
| `HANDOFF.md` | 熱區：當前狀態與最新進度 | 建議 100 行內 | 每次任務開始時必讀 |
| `HANDOFF_ARCHIVE.md` | 冷區：完整歷史紀錄 | **無上限，永不刪除** | 預設不讀，僅在使用者明確要求查歷史時讀取 |

### 標準格式(HANDOFF.md 熱區)

```markdown
## HANDOFF - [任務名稱]

### 本次完成事項
- ...

### 進行中 / 尚未完成
- ...

### 關鍵檔案清單
| 檔案路徑 | 用途說明 |
|---|---|
| path/to/file | 說明 |

### 注意事項 / 已知風險
- ...

### 下一步建議
- ...

### 關鍵決策紀錄
- ...
```

### 簡化版(對話中途快速存檔用)

```markdown
## HANDOFF - [任務名稱]
完成：...
注意：...
下步：...
```

### 歷史搬遷規定

當 HANDOFF.md 主檔過長時，**不可直接刪除**內容。必須：

1. 將要移出的段落**完整附加**到 `HANDOFF_ARCHIVE.md` 末尾
2. 加上日期戳與原任務名稱
3. 確認 ARCHIVE 寫入成功後，才從主檔移除
4. ARCHIVE 永遠只增不減

任何「壓縮」「摘要化」「精簡」操作，**前提都是先把原文搬進 ARCHIVE**。
未先歸檔就刪除歷史內容，視為 P0 違規。

## RULE 02 | 資源與成本回報 - Resource Accounting

任務完成時，最後必須回報：

```text
⏱️ 任務耗時：XX 分 XX 秒 | 🪙 Tokens (估算): IN XXk / OUT XXk | 💰 狀態: 成功/失敗/阻塞
```

### 補充規範

- 若有子任務，分別計時後再加總。
- 若任務中途中斷再續行，需區分本次耗時與累計耗時。
- 長時間任務（> 30 分鐘）應定期回報進度；若平台不支援穩定時間節點，至少在重要階段切換時回報。

### 數據真實性

若工具、平台、CLI 或執行環境無法可靠提供 token / 耗時估算，必須明確標示為下列之一：

- `估算`
- `不可得`
- `平台未提供`

**禁止事項**

- 不可編造虛假的 token、耗時或成本數字。
- 不可在無法取得實際數據時填入看似合理的猜測值。

## RULE 03 | 環境隔離 - Local-First Execution

所有執行與依賴優先放在本專案內，禁止污染 global 環境。

### 強制規範

- Python 使用 `.venv` 或專案專用 conda env。
- Node.js 套件只做 local install，禁止未授權的 global install。
- `.env` 放在專案內，並提供 `.env.example`。
- 禁止修改 shell profile 或系統全域設定。
- 優先使用專案內的 scripts、config、workspace。

### 建議目錄

```text
project_root/
├── .venv/
├── node_modules/
├── .env
├── .env.example
├── HANDOFF.md
├── HANDOFF_ARCHIVE.md
├── logs/
└── .gitignore
```

## RULE 04 | 版本快照 - State Snapshot

任何重要操作前，必須先建立可還原快照。

### 最低要求

- 記錄當前 `git HEAD`。
- 或輸出 patch 備份。
- 或備份關鍵檔案為 `.bak`。

### 破壞性操作前 Checklist

- [ ] 已建立備份或快照
- [ ] 已記錄 rollback 步驟
- [ ] 已評估影響範圍
- [ ] 已通知相關人員（若有）

### 例外情境

純閱讀、純審查、純回報、純進度查詢、純 prompt review 任務，若不涉及修改、刪除、重建、資料寫入，則不要求建立快照。

### 仍需快照的操作

只要涉及下列任一情境，仍視為重要操作，必須先做快照：

- 刪檔
- 重建
- 覆蓋
- 批次修改
- 資料表寫入
- 歷史結果處理
- HANDOFF.md 主檔搬遷至 ARCHIVE 之前

## RULE 05 | 最小權限與機密保護 - Least Privilege & Secret Masking

嚴禁在任何 log、handoff、對話、報表中明碼輸出下列資訊：

- API key
- token
- password
- private key
- credentials

### 遮罩範例

| 類型 | 範例 |
|---|---|
| API Key | `sk-proj-****1234` |
| Token | `ghp_****xyz` |
| Password | `****` |
| Private Key | `-----BEGIN PRIVATE KEY----- [REDACTED] -----END PRIVATE KEY-----` |

### 建議加入 `.gitignore` 的項目

```text
.env
*.key
*.pem
id_rsa*
*.log
*.tmp
*.cache
credentials.json
secrets.yaml
```

## RULE 06 | 明確溝通 - Explicit Communication

不確定、假設、風險、阻塞，不得靜默跳過。

### 強制標籤

| 標籤 | 使用情境 |
|---|---|
| `[ASSUMPTION]` | 做了未經確認的假設 |
| `[RISK]` | 有潛在風險的操作或決策 |
| `[BLOCKED]` | 被阻塞，需要人類介入 |
| `[SKIP]` | 主動略過某項並說明原因 |
| `[TODO]` | 應完成但本次未完成 |
| `[EXCEPTION]` | 申請規則例外並說明理由 |
| `[TENTATIVE]` | 暫定的決策或結論，可能調整 |

### 補充規範

- 回報內容若同時包含事實、推測、建議，必須分段標示，不可混寫。
- 不可把推測寫成既定事實。
- 不可把未驗證的內容寫成已完成。

## RULE 07 | 冪等性 - Idempotency

腳本與任務設計必須可安全重複執行。

### 強制規範

- 建立檔案前先檢查是否已存在或內容是否相同。
- 批次任務要避免重複寫入。
- 可重跑不代表可以覆蓋歷史資料。
- 若外部系統支援，優先使用 idempotency key。

## RULE 08 | 日誌義務 - Logging Standard

重要操作必須留下可追蹤日誌。

### 格式

```text
[YYYY-MM-DD HH:MM:SS] [LEVEL] [MODULE] Message
```

### Log Level

| Level | 用途 |
|---|---|
| INFO | 正常執行流程 |
| WARN | 非預期但不中斷流程 |
| ERROR | 需要處理的錯誤 |
| DEBUG | 開發除錯用，正式環境預設關閉 |

### 保留原則

- 單一日誌檔案不超過 100MB。
- 開發環境可短期保留，正式環境需有輪替策略。
- 重要失敗必須能對應到任務、模組與時間點。

### 例外情境

純閱讀、純審查、純規格 review 任務，不強制要求產生專案內日誌檔。
但即使不產生日誌檔，回報中仍需清楚說明判斷依據與結論來源。

## RULE 09 | 防卡死與重試限制 - Infinite Loop Prevention

同一個錯誤或卡點，自動重試不得超過 3 次。

### 決策規則

1. 第 1 至 2 次失敗：可在有新資訊或修正條件下重試。
2. 第 3 次失敗：必須標記 `[BLOCKED]`，停止該子任務。
3. 保留錯誤現場與必要 log，等待人類或上層 Agent 處理。
4. `[BLOCKED]` 一旦標記，必須立即寫入 `HANDOFF.md`(依 RULE 01 強制存檔義務)。

## RULE 10 | 自我驗證義務 - Self-Validation Requirement

未經驗證，不得宣稱完成。

### 基本驗證順序

1. 編譯或建置成功
2. Lint / 語法檢查通過
3. 單元測試通過
4. 整合測試通過（若有）
5. 手動驗證關鍵流程

### 變更類型對應驗證

| 變更類型 | 必要驗證 |
|---|---|
| UI | 視覺檢查、互動流程、響應式檢查 |
| API | 端點測試、錯誤處理、文件同步 |
| 資料庫 | 資料完整性、rollback、效能影響 |
| 設定 | 語法檢查、影響範圍確認 |
| 相依性更新 | 建置成功、相關測試通過 |

### 與 RULE 15 的互動

Review 任務的「完成」定義為：產出書面審查意見、找出問題、給出建議。
Review 任務不需透過實際執行來驗證，因此本規則的「自我驗證」在 review 情境下指：

- 審查邏輯自洽
- 引用準確
- 結論有依據

而非「跑通代碼」或「實際試跑」。

## RULE 11 | 資料流優先繼承 - Data Pipeline Reuse First

若專案已有既有資料流或資料工程基底，必須先檢查能否重用。

### 本專案硬規則

若使用者已指定資料流路徑，例如：

```text
C:\Users\User\Documents\data_pipeline
```

則必須先：

1. 掃描該路徑。
2. 找出可重用模組。
3. 優先複製必要元件到本專案內。

### 禁止事項

- 未檢查就假設資料格式。
- 繞過既有資料讀取邏輯重寫一套。
- 直接修改原始 `data_pipeline`，除非任務明確要求。

### 例外情境

本規則適用於任務確實涉及資料讀取、資料流接續、資料工程重用時。
若任務與資料流無關，應在回報中標註 `[SKIP]` 並說明原因。

## RULE 12 | PostgreSQL 歷史資料保護 - History Must Not Be Overwritten

凡屬歷史結果、策略紀錄、投組快照、訊號快照之資料表，不得默默覆蓋舊資料。

### 強制規範

- 優先使用 `INSERT ... ON CONFLICT DO NOTHING`。
- 禁止 `ON CONFLICT DO UPDATE` 直接覆寫歷史結果。

### 若需修正舊資料，只能二選一

1. 使用新的 `strategy_name` 或版本名稱重新寫入。
2. 人工明確 `DELETE` 後，再重新 `INSERT`。

## RULE 13 | 先做最小可行版 - MVP First

禁止一開始就過度複雜化。

### 建議順序

1. 先讓資料能正確讀進來。
2. 先讓回測主流程能跑完。
3. 先產出第一版結果。
4. 再做參數掃描、風控擴充、報表美化。

## RULE 14 | 結果真實性優先 - No Cosmetic Research

研究結果必須誠實呈現，不得為了好看而調整敘事。

### 強制規範

- 若策略結果差，就直接說差。
- 若 turnover 太高，就直接說不可交易。
- 若 funding 或成本吃光 edge，就直接說沒有研究價值。

### 禁止事項

- 偷改 universe 美化結果。
- 選擇性只展示好看的參數。
- 忽略成本。
- 隱藏失敗實驗。

## RULE 15 | Review 與 Execution 分離 - Review Mode Boundary

若使用者明確要求的是下列任一類型的任務：

- review
- 審查
- prompt review
- md review
- spec review
- handoff review
- 只讀分析

則不得將 review 任務偷換成 execution 任務。

### 強制規範

- 不可執行被審查任務本體。
- 不可部分執行。
- 不可 dry run。
- 不可為了「驗證可行性」而先做一部分。
- 不可以「輔助確認」為由執行任務本體。
- 不可把 review 變成任務試跑、環境驗證、局部執行或工作區修改。

### 模糊用語的處理

若使用者使用「評估」「分析」等可雙向解讀的詞，且同時要求驗證可行性、跑通、實際確認，
**Agent 必須在開始前明確詢問**：

> 「這是純 review，還是包含執行？」

不可自行假設為其中之一。

### Review 任務的完成定義

Review 任務的「完成」是：

- 產出書面審查意見
- 找出問題
- 給出建議

不需要透過實際執行來驗證。詳見 RULE 10「與 RULE 15 的互動」。

### 若已越界

若在 review 過程中已實際執行任務內容，必須：

1. 明確承認已越界。
2. 列出實際做了哪些事。
3. 不可再把這些執行結果當成 review 的主要依據。
4. 必須回到「尚未執行任務」的角度重新審查。

## Agent 任務完成 Checklist

- [ ] 已更新 `HANDOFF.md`(且符合 RULE 01 存檔時機)
- [ ] 必要時已將舊內容歸檔至 `HANDOFF_ARCHIVE.md`
- [ ] 已回報耗時與資源估算
- [ ] 未污染 global 環境
- [ ] 已建立快照或可回滾方案
- [ ] 無機密外洩
- [ ] 風險與假設已標示
- [ ] 任務可重跑
- [ ] 已留下必要日誌
- [ ] 未超限重試
- [ ] 已完成自我驗證
- [ ] 已檢查既有 data pipeline 是否可重用
- [ ] PostgreSQL 歷史表未被覆蓋
- [ ] 已誠實揭露策略真實表現
- [ ] 未越界執行 review 任務

### Checklist 補充說明

若任務屬於純閱讀、純審查、純 review，則下列項目可依實際情況標註 `[SKIP]`，但必須說明原因：

- 快照
- 專案內日誌
- data pipeline reuse
- PostgreSQL 寫入保護檢查

**不可 SKIP 的項目**(無論任務性質均適用)：

- HANDOFF 存檔義務(RULE 01)
- 資源回報
- 機密保護
- 風險標示
- Review 邊界
