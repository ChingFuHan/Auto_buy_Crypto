# AGENTS.md

> 本檔是**給 AI agent 看的專案進場手冊**。
> 通用規則見 `god_rule.md`,本檔只負責本專案專屬的進場順序與領域地雷。

---

## 上位規則

本專案的最高規則為 `god_rule.md`(v1.5+)。

- `god_rule.md` 優先級高於 `AGENTS.md`
- 若任何文件(`AGENTS.md`、`README.md`、`HANDOFF.md`、`final_file.md`、任務 prompt)與 `god_rule.md` 衝突,先列出衝突點,再以 `god_rule.md` 為準
- 不可忽略、弱化、覆寫 `god_rule.md`

---

## 進場閱讀順序

第一次進入本專案,依序讀:

1. `god_rule.md`
2. `README.md`
3. `HANDOFF.md`(熱區,當前進度)
4. `final_file.md`(僅在任務**明確提到** `final_files`、交付、匯出、封裝時)

> `HANDOFF_ARCHIVE.md` 預設**不讀**,僅在使用者明確要求查歷史時才讀取。

---

## 存檔義務(P0)

依 `god_rule.md` RULE 01,Agent 必須在以下時機**立即**寫入 `HANDOFF.md`:

- 子任務完成、關鍵決策做出、卡點解開
- 使用者出現「先這樣」「我先去 X」「明天繼續」等收尾訊號
- 長對話累積 20 輪以上
- Agent 自行判斷對話可能即將結束時

**對話視窗一旦關閉,未存檔內容永久消失**。延後存檔=P0 違規。

歷史搬遷規則(熱區轉冷區)、`[TENTATIVE]` 標籤等細節,完整規則見 `god_rule.md` RULE 01。

## 本專案領域規則(交易系統專屬)

本專案是交易系統。以下事項**未經明確要求,絕對不可自行執行**——錯一次可能造成真實虧損或不可逆破壞:

- 不可修改交易邏輯
- 不可切換 live / testnet 環境
- 不可建立新憑證
- 不可修改 `.env`
- 不可寫入帳戶或交易所 API
- 不可修改 PostgreSQL 歷史結果表(歷史資料保護見 `god_rule.md` RULE 12)

若任務依賴上述任一項,**只能使用現有可用設定**。條件不足時,先停下來回報。

## 主線 / 正式版判定

若主線、正式版、生效版本不明確,參考以下文件之間的一致性:

- `README.md`
- `HANDOFF.md`
- 主程式入口
- 核心模組
- 測試內容

仍不明確時,**停下來回報,不可自行假設**。

## 工作原則

- 先理解,再動手
- 先做小範圍、可追蹤的變更
- 修改範圍緊貼需求,不可順手重構無關部分
- 文件已足夠時,不要重掃整個專案
- 所有判斷區分:已確認事實 / 推測 / 未確認事項

### 不可自行新增的檔案

未經明確要求,**不可自行建立**:

- 新的規則文件
- 新的摘要文件
- 新的交接文件(`HANDOFF.md` 與 `HANDOFF_ARCHIVE.md` 是專案標準檔,不在此限)
- 臨時說明文件(如 `NOTES.md`、`PROJECT_SUMMARY.md`、`TEMP_PLAN.md`)

### 必須停下來回報的情境

- 主線 / 正式版不明確
- 任務指令與既有文件規則衝突(不可默默選一邊執行)
- 既有文件之間互相衝突(不可自行決定誰優先)
- 領域規則邊界不清(例:不確定某個改動是否動到交易邏輯)

## 查詢進度時

當使用者詢問目前進度、最近完成什麼、現在做到哪:

1. 先讀 `HANDOFF.md`(進度資訊密度最高)
2. 用 `README.md` 補背景
3. 文件不足時,才**最小範圍**檢查核心程式
4. 不要大範圍掃描專案
5. 不要修改任何檔案
6. **不可把進度查詢升級成完整專案重新閱讀**

> 注意:此處順序是「進度查詢專用」。第一次進場仍依「進場閱讀順序」章節。
> 若使用者要求查更早的歷史,才讀取 `HANDOFF_ARCHIVE.md`。

## Review 模式

若任務是 review / 審查 / 純分析類,**完整規則見 `god_rule.md` RULE 15**。

本專案的進場提醒:

- 在本專案中,「修改工作區、跑命令、連線 API、查帳戶、查資料庫」**都算執行,不算 review**
- 若使用者用「評估」「分析」等模糊詞,且看起來可能要動到上述項目,必須先問清楚是純 review 還是包含執行

## 檔案修改安全

修改前先區分:

- 哪些檔案在本次任務範圍內
- 哪些不在範圍內
- 哪些是正式版
- 哪些可能是舊版 / 備份 / 實驗稿 / 輸出物

若資料夾看起來像**輸出區、快照區、實驗區、暫存區、歷史結果區**,不可自行假設可以安全修改。

`HANDOFF_ARCHIVE.md` 屬於**永久保留檔**,只增不減,不可刪改既有內容(只可附加)。

刪除動作只限明確指定的目標,不可擴大,也不可刪除名稱相近的其他檔案或資料夾。

---

## 回報格式

回報時用簡潔、結構化格式,至少說清楚:

- 讀了哪些檔案
- 得出哪些結論
- 改了哪些檔案
- 刻意沒有改哪些部分
- 風險 / 阻塞 / 未確認事項

事實、推測、建議**分段標示**,不可混寫。

標籤(`[ASSUMPTION]` / `[RISK]` / `[BLOCKED]` / `[SKIP]` / `[TODO]` / `[EXCEPTION]` / `[TENTATIVE]`)、資源回報、快照、驗證等規則,**全部依 `god_rule.md` 為準**,本檔不重列。

---

## 不確定時的優先順序

1. 保持目前專案狀態不被破壞
2. 明確回報已知事實、推測與不確定點
3. 等待使用者明確指示
4. 不自行做大範圍假設
5. 不自行擴張任務範圍

---

## 最後原則

本專案以「**先理解、再執行、可追蹤交接**」為優先。

不確定就停下來說明,不要自作主張。

任何對話的成果,都必須及時寫進 `HANDOFF.md`——使用者來這裡是為了不必記得。

所有上位規則(例外、標籤、驗證、快照、資源回報、Review 邊界、存檔義務)以 `god_rule.md` 為準。
